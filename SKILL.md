# 接棒

## 定位

它不是知识库，也不是笔记管理器。它负责让 Agent 在多个项目之间正确接班、归档材料、读取现场、调度任务、回写结果。

一句话：

```text
让 Agent 每次不是从聊天框重新开始，而是接住项目上一棒。
```

## 默认工作区

如果用户没有指定目录，在当前工作目录创建：

```text
Agent_Workspace/
```

根目录结构：

```text
Agent_Workspace/
  00_Inbox/
  99_Shared_Materials/
  项目索引.md
  全局方法/
    写作方法.md
    选题判断.md
    审稿标准.md
    工具使用规则.md
  项目/
    项目A/
    项目B/
```

每个项目目录：

```text
项目名/
  00_项目现场.md
  01_判断记录.md
  02_完成标准.md
  03_素材索引.md
  04_复盘记录.md
  05_会议记录/
  06_原始材料/
  07_产出物/
  08_待办.md
  09_接班记录.md
```

## 核心职责

### 1. 项目识别

当用户给出任务或材料时，先判断它属于哪个项目。

识别优先级：

1. 用户明确指定。
2. 当前任务上下文。
3. 当前文件路径或项目目录。
4. `项目索引.md` 的项目名、别名、关键词、参与人。
5. 材料内容推断。

识别门禁：

- 命中一个项目且置信度高于 0.7，继续。
- 命中多个项目，输出候选项目、命中证据和置信度，不写入。
- 置信度低于 0.7，只问一个问题：`这是新项目，还是放到已有的哪个项目里？`
- 用户确认前，只允许写入 `00_Inbox/` 临时草稿，不允许写入项目目录。

不要在项目不明时直接归档或开写。

### 2. 项目现场管理

每个项目都必须能回答：

- 这个项目是干什么的？
- 当前做到哪一步？
- 下一步只做什么？
- 哪些判断已经确认？
- 哪些方向已经否掉？
- 什么结果才算完成？
- 核心素材在哪里？

如果项目目录不存在，先按模板创建基础文件，再继续任务。

### 3. 材料入库

当用户提供会议记录、网页、PDF、截图、聊天记录、访谈、链接或原始文本时，不要只总结。

先做四件事：

1. 判断项目归属。
2. 检查敏感信息。
3. 生成素材 ID 和文件名。
4. 决定保存原文、摘要、还是只保存链接。

材料分类：

| 类型 | 默认放置 |
| --- | --- |
| 会议记录 | `05_会议记录/` |
| 网页/PDF/截图/原始文本 | `06_原始材料/` |
| 用户反馈/访谈 | `06_原始材料/用户反馈/` |
| 竞品/外部案例 | `06_原始材料/外部参考/` |
| 最终稿/方案/页面/报告 | `07_产出物/` |
| 新判断/否定方向 | `01_判断记录.md` |
| 可复用方法 | `全局方法/` 或项目内方法段 |

命名规则：

```text
原始材料：YYYY-MM-DD_类型_来源_短标题_v01.ext
会议记录：YYYY-MM-DD_会议_主题_参与方_v01.md
产出物：YYYY-MM-DD_产出类型_项目名_v01.md
```

禁止使用：

- `最终版`
- `最新版`
- `新建文档`
- 无日期文件名
- 只用人名、手机号、公司名、金额做文件名

同名文件必须递增 `v02`，不得覆盖。

去重规则：

- 同 URL、同文件路径、同内容 hash、同标题+同日期，视为疑似重复。
- 疑似重复时先更新已有索引的版本或备注，不新建材料。
- 无法判断是否重复时，写入 `00_Inbox/`，等待用户确认。

入库后必须更新 `03_素材索引.md`，记录：

- 素材名称
- 来源/链接/文件路径
- 素材 ID 或内容 hash
- 日期
- 类型
- 属于哪个问题
- 可用价值
- 是否核心证据
- 是否仍有效
- 敏感级别

### 4. 开工前调度

执行任何非简单任务前，先读取对应项目的：

```text
00_项目现场.md
01_判断记录.md
02_完成标准.md
03_素材索引.md
08_待办.md
09_接班记录.md
```

再按任务类型读取全局方法文件。

开工前输出一个简短调度摘要：

```text
识别项目：
当前阶段：
本次任务：
需要继承的判断：
不能再走的方向：
本次完成标准：
准备使用的材料/方法：
```

如果用户只要快速执行，可以把摘要压缩到 3 行。

### 5. 任务执行

中控不替代具体能力。它只负责把任务交给正确上下文。

常见任务路由：

- 写文章/方案：读取项目现场、判断记录、素材索引、写作方法、完成标准。
- 整理会议：保留原文，提取决策、待办、风险、可复用判断。
- 做选题：读取被否方向、目标用户、判断尺、素材索引。
- 做报告/交付：读取完成标准、产出物、证据素材、待办。
- 做复盘：读取项目现场、最近产出、失败记录、用户反馈。

### 6. 结束后回写

任务完成后，判断是否需要回写。

默认不要直接回写长期文件。先输出回写计划，除非用户已经明确授权“直接写入/保存/更新”。

回写计划必须包含：

- 拟写入文件
- 写入类型：事实 / 判断 / 待办 / 方法 / 产出
- 证据来源
- 是否覆盖旧内容
- 是否需要用户确认

需要回写的内容：

- 项目进度变化 -> `00_项目现场.md`
- 新确认判断 -> `01_判断记录.md`
- 新否定方向 -> `01_判断记录.md`
- 新素材 -> `03_素材索引.md`
- 新产出 -> `07_产出物/`
- 新待办 -> `08_待办.md`
- 复盘和踩坑 -> `04_复盘记录.md`
- 可跨项目复用的方法 -> `全局方法/`
- 下次接班信息 -> `09_接班记录.md`

### 6.1 自动沉淀规则

自动沉淀不是把对话全存下来，而是把下一次接班如果不知道就会返工的东西写进去。

只要任务中出现下面内容，默认进入回写计划；如果用户已经明确授权“直接写入/保存/更新”，则直接追加到对应项目文件：

| 内容类型 | 触发条件 | 写入位置 | 写入要求 |
| --- | --- | --- | --- |
| 新进度 | 项目阶段变化、完成了某个动作、验证通过/失败、交付物已生成 | `00_项目现场.md` 的最近进展；必要时同步 `09_接班记录.md` | 写清日期、动作、结果，不写过程废话 |
| 结论性判断 | 出现“所以现在应该/不能/优先/暂缓/砍掉/继续”的明确取舍 | `01_判断记录.md` 的已确认判断或待验证判断 | 必须写依据、影响、复核日期、失效条件 |
| 被否方向 | 明确发现某条路不该走、成本过高、证据不足或已失败 | `01_判断记录.md` 的已否定方向 | 写清为什么不行，以及以后怎么避免再走 |
| 新待办 | 出现明确下一步，且有可执行对象或产出 | `08_待办.md`；必要时同步 `09_接班记录.md` | 写优先级、任务、阶段、状态和备注 |
| 新素材 | 用户给了会议、链接、文件、截图、聊天、调研、反馈、报告 | 原文放 `05_会议记录/` 或 `06_原始材料/`，索引写 `03_素材索引.md` | 保留来源、日期、类型、价值、敏感级别 |
| 新产出 | 生成了方案、报告、页面、文案、清单、模板、脚本 | `07_产出物/`，并在 `03_素材索引.md` 记录 | 文件名带日期和版本，不用“最终版” |
| 踩坑复盘 | 发生失败、返工、误判、验证不通过、流程卡住 | `04_复盘记录.md` | 写发生了什么、原因、影响、下次避免方式 |
| 接班信息 | 任务结束时存在未完成动作、关键判断或禁止方向 | `09_接班记录.md` | 写当前唯一下一步、必须继承的判断、不能再走的方向 |

下面这些默认不沉淀，只在用户明确要求时写入：

- 临时聊天、情绪表达、无行动价值的吐槽。
- 没有证据的猜测。
- 只适用于当前一次对话、下次不会复用的细节。
- 还没被验证的灵感，除非写入“待验证判断”。
- 项目私有经验，不直接写入 `全局方法/`。

沉淀优先级：

1. 会影响下一步动作的结论，必须沉淀。
2. 会避免下次返工的踩坑，必须沉淀。
3. 会改变项目阶段的进度，必须沉淀。
4. 只是解释背景、但不影响后续动作的内容，不沉淀。

直接写入前必须检查：

- 是否知道归属项目。
- 是否会覆盖旧内容。原则上只追加，不覆盖。
- 是否含敏感信息。
- 是否是事实、判断、待办、方法还是噪音。
- 是否需要写入多个文件保持一致，例如进度写 `00_项目现场.md`，下一步同步 `09_接班记录.md`。

回写原则：

- 不覆盖旧判断，只追加带日期的新条目。
- 不经确认，不写入 `全局方法/`。
- 临时想法不写成长期规则。
- 用户明确否定的内容必须记录到“已否定方向”，但不能扩大成无关项目规则。

回写前要区分：

- **事实**：真实发生、可溯源。
- **判断**：基于事实形成的取舍。
- **方法**：以后可以复用的流程或标准。
- **待办**：下一步动作。
- **噪音**：暂不沉淀。

### 7. 隐私与对外使用红线

默认所有原始材料都是 private。

入库前检查：

- 手机号、邮箱、身份证件、家庭地址。
- 合同金额、客户名、内部报价、商业机密。
- 内部链接、聊天截图、会议录音。
- token、password、API key、cookie、私钥。

处理规则：

- 发现 token/password/API key/private key，停止入库，只提示用户先清理。
- 含敏感信息的材料，文件名不得包含人名、公司名、手机号、金额。
- 对外导出只使用脱敏摘要，不直接复制原文。
- 跨项目引用敏感材料时，只写摘要和指针，不复制原文。
- 用户明确要求删除或不保存原文时，只保存摘要和来源说明。

## 工作流

### 新项目

1. 询问或识别项目名、目标、当前阶段。
2. 创建项目目录和基础文件。
3. 更新 `项目索引.md`。
4. 写入初始项目现场、完成标准、待办。
5. 写入 `09_接班记录.md`。
6. 进入任务。

### 继续旧项目

1. 识别项目。
2. 读取项目现场。
3. 给出开工调度摘要。
4. 执行任务。
5. 回写进度和判断。

### 整理材料

1. 判断材料归属项目。
2. 保存原始材料。
3. 提取摘要、决策、风险、待办、可用证据。
4. 更新素材索引。
5. 如产生新判断，同步写入判断记录。

### 多项目冲突

如果一个材料同时涉及多个项目：

1. 优先保存到 `99_Shared_Materials/`，生成唯一素材 ID。
2. 相关项目的 `03_素材索引.md` 写引用和引用原因。
3. 敏感材料默认不跨项目引用。
4. 不要复制多份造成版本不一致。

交叉引用字段：

```text
canonical_id / 原件路径 / 可见项目 / 引用原因 / 敏感级别 / 摘要版路径
```

### 失败条件

以下情况停止写入，只输出待确认清单：

- 项目归属冲突。
- 同名项目或同名文件冲突。
- 原始材料含高敏感信息。
- 无法保存原文。
- 需要覆盖已有文件。
- 要写入 `全局方法/`。
- 工作区目录不可写。

## 输出原则

- 先判断项目归属，再处理内容。
- 保留原始材料，不只保存摘要。
- 先提出回写计划，再写长期文件。
- 不把所有东西都写进记忆。
- 不把临时想法写成长期规则。
- 不把旧判断当成当前标准。
- 不把项目私有判断沉淀成全局方法，除非它至少跨 2 个项目复用，并且有输入、输出和失败边界。

## 可用脚本

优先用脚本完成重复动作：

```text
scripts/init_workspace.py --root Agent_Workspace
scripts/create_project.py --root Agent_Workspace --name "项目名"
scripts/validate_workspace.py --root Agent_Workspace
```

脚本只负责创建和验证结构，不替代内容判断。

## 验收

创建或更新后必须检查：

- 工作区目录存在。
- `项目索引.md` 存在且有项目记录。
- 项目目录存在。
- 核心文件非空。
- 素材入库后 `03_素材索引.md` 有新增或更新行。
- 任务结束后输出新增/更新文件清单、未回写项、下次启动建议。

## 何时读取模板

需要创建项目文件时，读取 `templates/project_files.md`。

需要生成启动或回写提示词时，读取 `templates/prompts.md`。
